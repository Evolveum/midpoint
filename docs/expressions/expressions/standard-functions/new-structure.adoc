= MidPoint standard functions
:page-toc: top
:toclevels: 1
:page-nav-title: Standard functions
:page-keywords: function, midpoint functions, function library, script, script function, groovy,
:page-description: The built-in midPoint functions expose all core operations, such as object creation, retrieval, or invitation link generation to expression scripts.

The `MidpointFunctions` interface defines the built‑in midPoint functions that you can use in scripts.
These functions expose the core model operations (object creation, retrieval, modification, search, workflow, cryptography, etc.) as a concise, script‑friendly API.
All functions are public members of the interface;
they can be invoked directly from an expression code (e.g., Groovy, JavaScript, etc.) without importing any additional classes.

*The functions are available in link:https://github.com/Evolveum/midpoint/blob/master/model/model-api/src/main/java/com/evolveum/midpoint/model/api/expr/MidpointFunctions.java[MidpointFunctions.java]*. +
The link leads to the `master` branch.
Select the branch that corresponds to your midPoint version, e.g., `support-4.10`.

[NOTE]
====
*Notes on generic placeholders and overloads*

* Generic type parameters (`<T>` and `<F>`)
    ** `<T extends ObjectType>` is used for functions that operate on any concrete midPoint object type (e.g., `UserType`, `RoleType`, `OrgType`).
        The caller supplies the concrete class via a `Class<T>` argument, and the function returns a value of that same type, preserving compile‑time type safety.
    ** `<F extends FocusType>` is a specialization of `<T>` for focus objects (objects that can have assignments, projections, and a lifecycle), such as users, org units, and services.
        Functions that manipulate focus‑specific data (e.g., `recompute`, `isDirectlyAssigned`) use `<F>` to signal that the operation is only applicable to focus types.
    ** The placeholder name (`T` or `F`) is irrelevant to the runtime;
        it merely conveys the intended category of objects and enables the compiler to enforce correct usage.
* Var‑args (`...`) versus collection overloads
    ** Functions that accept a var‑args parameter (e.g., `executeChanges(ObjectDelta<? extends ObjectType>... deltas)`) allow callers to pass any number of arguments directly, without constructing a collection first.
        Internally the compiler creates an array, so the call is convenient for a small, known number of deltas.
    ** The corresponding collection overload (`executeChanges(Collection<ObjectDelta<? extends ObjectType>> deltas)`) is intended for scenarios where the deltas are already assembled in a `List`, `Set`, or another `Collection`. It avoids the extra array allocation that the var‑args version would incur and is more appropriate for bulk or dynamically built delta sets.
    ** Both overloads perform the same operation; the choice is purely a matter of API ergonomics and performance considerations. The var‑args version is annotated with `@SuppressWarnings("unchecked")` because Java cannot create a generic array at runtime, while the collection version does not have that limitation.

== Create and change objects

=== createEmptyObject()

The `createEmptyObject` and `createEmptyObjectWithName` functions are used to construct Java instances of an object before adding them to the repository.
Note that these functions construct the objects *in memory only*.
If you intend to create an object in the repository or on a resource, you need to populate this object with data and then invoke the link:#addobject[`addObject()`] or link:#executechanges[`executeChanges()`] functions.

Creates an empty Prism object of the given type in memory.

* parameters:
    ** `Class<T> type`
* returns:
    ** `T`
* throws:
    ** `SchemaException`

=== searchObjects()

Searches for objects of the given type using the supplied query.

Returns an empty list if the object type is correct but no objects of that type exist.
Fails if the object type is wrong.

This function has a very limited scaling capability as all the results are stored in memory.
DO NOT USE on large datasets.
Recommended usage is only when using queries that cannot return a large number of results (e.g., queries for unique values) or when combined with paging capability.
For other cases, use link:#searchobjectsiterative[`searchObjectsIterative()`].

Use `options` to influence retrieval and processing of the objects.

* parameters:
    ** `Class<T> type, ObjectQuery query`
    ** `Class<T> type, ObjectQuery query, Collection<SelectorOptions<GetOperationOptions>> options`
    ** `TypedQuery<T> query`
    ** `TypedQuery<T> query, Collection<SelectorOptions<GetOperationOptions>> options`
* returns:
    ** `List<T>`
* throws:
    ** `CommunicationException`
    ** `ConfigurationException`
    ** `ExpressionEvaluationException`
    ** `ObjectNotFoundException`
    ** `SchemaException`
    ** `SecurityViolationException`

=== executeChanges()

Executes changes using the provided object deltas.
The order of execution is not defined and the implementation is free to determine the correct or most suitable ordering.
All deltas must relate to analogous objects (e.g., user and linked accounts).
Only one delta per object is accepted.
The implementation may throw an error if the objects are not analogous.
The implementation also implicitly links the objects if such a link is part of the data model.
For example, the function links all accounts to the user if they are passed in a single delta collection.
This is especially useful if the account deltas are ADD deltas without OID and therefore cannot be linked explicitly.
The OID provided in ADD deltas should be empty under usual circumstances.

Use `options` to influence processing of the deltas.

* parameters:
    ** `Collection<ObjectDelta<? extends ObjectType>> deltas`
    ** `Collection<ObjectDelta<? extends ObjectType>> deltas, ModelExecuteOptions options`
    ** `ObjectDelta<? extends ObjectType>... deltas`
* returns:
    ** `void`
* throws:
    ** `CommunicationException`
    ** `ConfigurationException`
    ** `ExpressionEvaluationException`
    ** `ObjectAlreadyExistsException`
    ** `ObjectNotFoundException`
    ** `PolicyViolationException`
    ** `SchemaException`
    ** `SecurityViolationException`

=== isEvaluateNew()

Returns indication whether a script code is evaluating a new value.
If this method returns `true` value, then the script code is evaluating a "new" value.
That means a value that is going to be set to target, e.g., a value from the "add" or "replace" parts of a delta.
If the script evaluates existing value that is not being modified (e.g., during recomputation), that is also considered as "new" value (going to be set to the target).
If this method returns `false` value, then the script code is evaluating an "old" value.
That means a value that is used for the purposes of removing values from target, e.g., a value from the "delete" part of a delta or values from an existing object.
